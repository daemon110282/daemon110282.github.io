# Локализация i18n

## Зачем

- Избегайте применения жестко кодированных строк и внешних ресурсов для текстовых данных или данных компоновки (например, для поддержки языков с написанием справа налево), особенно если приложение будет подлежать локализации
- Локализация - текстов сообщений (resource) с бэка
- Останется только:
  - на бэке по БД формировать json в нужном формате, которые примет vue-i18n
  - или же вообще [вынести эту функцию из ИС](https://www.codeandweb.com/babeledit)

## Паттерны

- Ключи должны быть:
  - стабильными (не зависят от текста)
  - иерархическими (screen.section.element)
- Разделять ресурсы:
  - по фичам (предпочтительно)
  - или по экранам
- Работа с дизайном
  - Текст может увеличиваться на 30–200%
  - переносы строк
  - multiline
- ICU (International Components for Unicode)
  - набор библиотек и стандартов для интернационализации (i18n), который решает ключевые задачи локализации: форматирование дат, чисел, валют, работа с Unicode и, самое важное — MessageFormat для переводов с логикой
  - “движок”, который позволяет не просто хранить переводы, а грамотно адаптировать текст под язык и контекст
- RTL (Right-to-Left) поддержка
- Pluralization
- Динамическая загрузка переводов
  - Lazy loading
  - OTA
- Управление переводами - Translate Managment System (TMS)
- CI проверки переводов
